# 本当のMVP提案

## 現在の問題

現在の仕様 = **MVP（最小限）ではなくMFP（最大限）**

```yaml
現状のPhase 1:
  機能数: 19個
  開発期間想定: 4-5週間
  現実の開発期間: 8-12週間
  成功確率: 10%以下
```

## 真のMVPの定義

**「核心価値を検証できる最小限の機能のみ」**

あなたのアプリの核心価値は何？
```yaml
核心価値:
  「他人の作業風景を見て、自分も頑張れる」

これを検証するために本当に必要なのは:
  ✅ 動画を投稿できる
  ✅ 動画を見られる

以上。
```

## このミニマムMVPで検証できること

```yaml
核心的な質問:
  1. ユーザーは動画を投稿するのか？
     → 投稿数を測定

  2. ユーザーは他人の動画を見るのか？
     → 再生回数を測定

  3. ユーザーは継続して使うのか？
     → DAU/MAUを測定

これらが分かれば、次に進むか判断できる。
```

## 検証後の判断基準

### シナリオA: 成功（次のステップへ）
```yaml
2週間後の結果:
  - βテスター20人
  - 投稿数: 50本以上
  - 1人あたり平均2.5本投稿
  - 再生回数: 500回以上
  - 7日継続率: 30%以上

判断: ✅ 需要あり → Phase 1.5へ
Phase 1.5で追加:
  - いいね機能
  - カテゴリ
  - マイページ
  - 通知
```

### シナリオB: 微妙（ピボット）
```yaml
2週間後の結果:
  - βテスター20人
  - 投稿数: 10-20本
  - 1人あたり平均0.5-1本
  - 再生回数: 100-200回
  - 7日継続率: 10-20%

判断: ⚠️ 需要は微妙
分析:
  - 投稿ハードルが高い？ → 写真に変更検討
  - 視聴がつまらない？ → コンテンツ質の問題
  - コンセプトが伝わってない？ → 説明改善

ピボット案:
  1. 動画 → 静止画に変更
  2. 作業風景 → 作業環境（デスク写真）
  3. ランダム → フォロー制に変更
```

### シナリオC: 失敗（別アイデアへ）
```yaml
2週間後の結果:
  - βテスター20人
  - 投稿数: 5本以下
  - 1人あたり平均0.25本未満
  - 再生回数: 50回未満
  - 7日継続率: 5%以下

判断: ❌ 需要なし
学び:
  - 作業風景シェアのニーズは低い
  - 動画投稿のハードル高すぎ
  - 価値提案が弱い

次のアクション:
  1. 完全に別のアイデアを試す
  2. または就職して経験積む
  3. スキルは残ったので無駄ではない
```

## 最大の問題: 「あれもこれも症候群」

### なぜ機能を削れないのか？

```yaml
心理:
  「完璧じゃないとリリースできない」
  「この機能がないと使われない」
  「競合にはこの機能がある」

現実:
  → 完璧を目指して永遠に完成しない
  → 複雑すぎて誰も使わない
  → 競合の真似では差別化できない
```

## 提案: 超ミニマムMVP（2週間）

### 最小機能セット

```yaml
Week 1:
  Day 1-2: 環境構築
  Day 3-4: 認証
  Day 5-7: カメラ撮影 + アップロード

Week 2:
  Day 8-10: 一覧表示 + 再生
  Day 11-12: 削除機能
  Day 13-14: テスト
```

### 成功の定義

```yaml
2週間後:
  ✅ 動作するアプリ完成
  ✅ βテスター20人招待
  ✅ 投稿数50本以上
  ✅ 需要の有無が判明

次のステップ:
  需要あり → Phase 1.5で機能追加
  需要なし → 2週間で損切り、次へ
```

### この方法のメリット

```yaml
✅ 2週間で完成（モチベーション維持）
✅ 失敗しても傷浅い
✅ 需要検証が速い
✅ ピボット（方向転換）しやすい
✅ 完璧主義の罠を回避
```
